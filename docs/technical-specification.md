# Техническое задание

## 1. Название проекта

Система синхронного сбора, обработки и записи данных с двух измерительных трактов лазерной энергии.

## 2. Назначение

Разработать приложение под Windows с простым графическим интерфейсом для:

- одновременного сбора данных энергии с двух источников измерений;
- синхронизации измерений по времени срабатывания лазерного триггера;
- расчета усредненных значений по последним `N` срабатываниям;
- автоматического определения стационарного режима;
- фиксации результатов стационарного режима;
- возобновления контроля при выходе из стационарного режима;
- записи результатов и сырых измерений в файл формата `XLSX`;
- уведомления оператора при ошибках и диагностических событиях.

## 3. Исходные данные и оборудование

### 3.1. Измерительные тракты

Система использует два независимых измерительных источника:

- тракт 1: прибор Beam Gage, подключенный по USB;
- тракт 2: пироэлектрический датчик Ophir, подключенный через Pulsar-4.

### 3.2. Режим работы

- оба измерительных тракта работают по внешнему триггеру;
- источник триггера: лазер;
- измеряемая величина: энергия;
- ожидаемая частота опроса: до `1000 Гц`;
- длительность одной серии измерений: до `10 минут`.

### 3.3. Программная среда

- основная целевая операционная система: Windows 10/11;
- дополнительная совместимость: Windows 7 желательна, но не обязательна;
- версии драйверов и ПО уточняются позднее;
- документация, приложенная к архиву, считается соответствующей используемым версиям программ и драйверов.

## 4. Цели разработки

Приложение должно обеспечить единый операторский сценарий:

1. оператор запускает приложение;
2. задает параметры измерения;
3. подключает два измерительных тракта;
4. запускает сбор данных;
5. приложение непрерывно принимает измерения с двух трактов;
6. приложение синхронизирует и обрабатывает данные;
7. приложение определяет вход в стационарный режим;
8. приложение фиксирует результат стационарного режима;
9. при резком изменении значений приложение обнаруживает выход из стационарного режима и возобновляет контроль;
10. по завершении работы приложение сохраняет данные в `XLSX`.

## 5. Функциональные требования

### 5.1. Подключение и инициализация

Приложение должно:

- обнаруживать и инициализировать оба измерительных тракта;
- отображать оператору состояние подключения каждого тракта;
- выполнять проверку готовности перед стартом измерений;
- запрещать запуск серии, если хотя бы один из трактов не готов к работе.

### 5.2. Сбор данных

Приложение должно:

- одновременно принимать данные с двух трактов;
- получать энергию для каждого срабатывания;
- привязывать измерения к единой временной шкале;
- поддерживать непрерывную работу в течение серии до `10 минут`;
- корректно работать при частоте поступления данных до `1000 Гц`.

### 5.3. Синхронизация данных

Приложение должно:

- сопоставлять измерения двух трактов между собой по времени или по событию срабатывания;
- считать измерения синхронными, если они относятся к одному и тому же импульсу или попадают в допустимое окно рассогласования;
- фиксировать факт рассинхронизации, если измерения не удалось надежно сопоставить в пределах допустимого окна;
- сохранять время возникновения рассинхронизации.

Примечание:

Допустимое окно рассогласования должно быть параметром конфигурации или должно быть согласовано отдельно до начала реализации.

### 5.4. Обработка измерений

Для каждого измерительного тракта приложение должно:

- хранить буфер последних `N` измерений;
- рассчитывать среднее значение энергии по последним `N` срабатываниям;
- обновлять расчет при поступлении каждого нового значения;
- использовать пользовательское значение `N`.

### 5.5. Определение стационарного режима

Приложение должно:

- автоматически определять, что процесс вышел на стационарный режим;
- использовать настраиваемый порог чувствительности;
- использовать одну из метрик отклонения, выбранную разработчиком;
- не требовать от пользователя выбора конкретной метрики, если это не потребуется в ходе реализации.

Допустимая интерпретация стационарного режима:

- изменения усредненного значения на последних окнах малы;
- разброс значений на последних `N` срабатываниях не превышает заданного порога;
- система считается стационарной до тех пор, пока измерения остаются внутри установленного критерия.

### 5.6. Фиксация результата и повторный вход в измерение

Приложение должно:

- фиксировать результат в момент входа в стационарный режим;
- продолжать наблюдение за потоком данных после фиксации;
- определять резкое изменение значений как выход из стационарного режима;
- при выходе из стационарного режима возобновлять процесс контроля и ожидания нового стационарного участка;
- фиксировать повторные входы в стационарный режим в течение одной серии.

### 5.7. Экспорт данных

Приложение должно сохранять данные в файл `XLSX`.

В файл должны записываться:

- временная метка измерения;
- идентификатор импульса или порядковый номер срабатывания;
- значение энергии по тракту 1;
- значение энергии по тракту 2;
- усредненное значение по тракту 1;
- усредненное значение по тракту 2;
- признак синхронной пары измерений;
- события входа в стационарный режим;
- события выхода из стационарного режима;
- время рассинхронизации, если оно возникало;
- диагностические и аварийные события.

В файл не требуется записывать:

- статус датчика как отдельное измерительное поле.

Рекомендуемая структура `XLSX`:

- лист `RawData`: поток измерений;
- лист `Events`: события стационарности, рассинхронизации и ошибок;
- лист `Summary`: итог по серии измерений.

### 5.8. Пользовательский интерфейс

Требуется простейший графический интерфейс.

Интерфейс должен содержать:

- кнопку запуска измерения;
- кнопку остановки измерения;
- поле задания `N`;
- поле задания порога стационарности;
- поле задания допустимого окна синхронизации;
- индикаторы состояния подключения каждого тракта;
- индикатор текущего состояния: ожидание / измерение / стационарный режим / ошибка;
- отображение текущих значений энергии по обоим трактам;
- отображение текущих усредненных значений;
- журнал событий и ошибок;
- поле выбора пути сохранения выходного файла.

### 5.9. Уведомление оператора

Приложение должно уведомлять оператора о следующих ситуациях.

Критические ошибки:

- обрыв USB;
- отсутствие ответа от датчика или прибора;
- отказ инициализации измерительного тракта;
- невозможность продолжать сбор данных.

Реакция на критическую ошибку:

- немедленная остановка измерения;
- визуальное уведомление оператора;
- запись события в журнал приложения;
- запись события в `XLSX`, если файл уже открыт и доступен.

Некритические диагностические события:

- рассинхронизация между двумя каналами;
- вход в стационарный режим;
- выход из стационарного режима;
- завершение серии.

Отдельное условие:

- пропуск импульса не считается ошибкой.

## 6. Нефункциональные требования

### 6.1. Производительность

Система должна обеспечивать прием и обработку данных при частоте до `1000 Гц` без потери устойчивости интерфейса и без зависания приложения.

### 6.2. Надежность

Система должна:

- корректно завершать измерение при критических ошибках;
- не терять уже зафиксированные данные при штатной остановке;
- вести журнал событий для последующего анализа.

### 6.3. Удобство эксплуатации

Система должна быть ориентирована на оператора без необходимости работать вручную с исходным кодом, командной строкой или внутренними диагностическими инструментами.

### 6.4. Совместимость по ОС

Система должна в приоритетном порядке поддерживать:

- Windows 10;
- Windows 11.

Дополнительно допускается запуск на Windows 7, если это позволит используемый набор драйверов, COM-библиотек и vendor SDK.

Примечание:

Поддержка Windows 7 не является обязательным критерием первой версии и подтверждается только после практической проверки на рабочем окружении.

### 6.5. Расширяемость

Архитектура должна допускать:

- замену логики определения стационарного режима;
- изменение формата итогового отчета;
- расширение количества диагностических метрик;
- возможное добавление новых источников данных в будущем.

## 7. Логика обработки данных

### 7.1. Общий цикл

При старте измерения приложение:

1. инициализирует оба тракта;
2. очищает внутренние буферы;
3. запускает прием данных;
4. при поступлении каждого нового измерения пытается найти синхронную пару;
5. обновляет буферы последних `N` измерений;
6. пересчитывает усредненные значения;
7. проверяет критерий стационарности;
8. генерирует события входа или выхода из стационарного режима;
9. пишет данные и события в память и/или на диск.

### 7.2. Буферизация

Для каждого тракта должен поддерживаться кольцевой буфер последних `N` значений.

Буфер должен использоваться для:

- расчета текущего среднего;
- расчета метрики отклонения;
- подтверждения входа в стационарный режим;
- подтверждения выхода из стационарного режима.

### 7.3. Критерий стационарности

Поскольку конкретная метрика не принципиальна, в базовой версии допускается следующий подход:

- вычислять среднее по последним `N` значениям;
- вычислять относительное или абсолютное отклонение текущего окна от предыдущего окна;
- считать режим стационарным, если отклонение меньше пользовательского порога;
- считать режим нарушенным, если отклонение становится больше пользовательского порога или отдельного порога выхода.

Рекомендуется предусмотреть в коде возможность легко заменить этот критерий.

## 8. Состав выходных данных

### 8.1. Данные по каждому измерению

Каждая запись потока должна включать:

- локальную временную метку;
- порядковый номер события;
- значение энергии Beam Gage;
- значение энергии Ophir / Pulsar-4;
- среднее Beam Gage по последним `N`;
- среднее Ophir / Pulsar-4 по последним `N`;
- служебный признак, была ли найдена синхронная пара.

### 8.2. События

События должны включать:

- тип события;
- время события;
- текстовое описание;
- при необходимости числовой контекст: порог, значение отклонения, номер импульса.

## 9. Предполагаемая архитектура решения

Рекомендуемая архитектура:

- модуль работы с Beam Gage;
- модуль работы с Ophir / Pulsar-4;
- модуль синхронизации потоков;
- модуль расчета скользящих средних и стационарности;
- модуль сохранения в `XLSX`;
- модуль UI;
- модуль журнала событий и ошибок.

Рекомендуемые принципы:

- прием данных отделен от UI;
- обработка данных отделена от драйверного слоя;
- экспорт в файл отделен от онлайн-обработки;
- логика стационарности оформлена как отдельный заменяемый компонент.

## 10. Обработка ошибок

### 10.1. Критические ошибки

К критическим относятся:

- потеря связи по USB;
- отсутствие ответа от одного из устройств;
- отказ инициализации;
- сбой, делающий невозможным продолжение достоверного измерения.

При критической ошибке приложение должно:

1. остановить прием данных;
2. показать оператору сообщение;
3. зафиксировать ошибку в журнале;
4. завершить текущую серию как аварийно прерванную.

### 10.2. Рассинхронизация

Рассинхронизация не обязательно является критической ошибкой, но должна быть явно зафиксирована.

Приложение должно:

- зарегистрировать момент рассинхронизации;
- показать уведомление оператору;
- продолжить работу или остановиться в зависимости от будущего согласованного правила.

Примечание:

Политика остановки при длительной рассинхронизации требует отдельного подтверждения.

### 10.3. Пропуск импульса

Пропуск импульса:

- не считается ошибкой;
- может быть отражен в журнале как информационное событие, если это упростит диагностику.

## 11. Параметры, настраиваемые пользователем

Пользователь должен иметь возможность задавать:

- `N` для усреднения;
- порог стационарности;
- допустимое окно синхронизации;
- путь сохранения файла `XLSX`;
- имя серии или комментарий к измерению.

Дополнительно допускается:

- настройка порога выхода из стационарного режима отдельно от порога входа;
- настройка длительности подтверждения стационарного режима;
- настройка реакции на рассинхронизацию.

## 12. Ограничения и допущения

- версии драйверов и библиотек будут уточнены позднее;
- архивная документация считается актуальной для используемой инсталляции;
- гарантированная целевая платформа первой версии: Windows 10/11;
- Windows 7 рассматривается как дополнительный совместимый контур без обязательства полной поддержки на старте;
- конкретная библиотека для генерации `XLSX` будет выбрана на этапе реализации;
- допустимая метрика стационарности выбирается разработчиком, так как она не зафиксирована заранее;
- точный критерий синхронности должен быть подтвержден до реализации промышленной версии.

## 13. Вопросы, требующие дополнительного подтверждения

Перед началом полноценной реализации желательно подтвердить:

- точные модели оборудования и датчиков;
- допустимое временное рассогласование между трактами;
- нужно ли сохранять каждую точку потока или возможна частичная агрегация;
- нужно ли автоматически закрывать файл после каждой серии;
- требуется ли повторный запуск новой серии без перезапуска приложения;
- нужна ли калибровочная информация в интерфейсе или в отчете.

## 14. Критерии приемки

Система считается соответствующей ТЗ, если:

- оба тракта подключаются и начинают измерение из одного приложения;
- данные с двух трактов поступают в ходе одной серии длительностью до `10 минут`;
- приложение считает скользящие средние по последним `N` измерениям;
- приложение определяет вход в стационарный режим;
- приложение фиксирует выход из стационарного режима при резком изменении значений;
- приложение пишет поток измерений и события в `XLSX`;
- приложение показывает оператору критические ошибки;
- при критической ошибке приложение завершает серию;
- при рассинхронизации приложение фиксирует время рассинхронизации;
- пропуск импульса не приводит к аварийной остановке.

## 15. Предлагаемые артефакты реализации

На этапе разработки рекомендуется подготовить:

- прототип приложения с подключением к двум источникам;
- тестовый режим без реального оборудования при возможности;
- пример выходного `XLSX`;
- журнал событий для диагностики;
- краткую инструкцию оператора.
