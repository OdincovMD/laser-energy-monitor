# Технический стек и архитектура

## 1. Краткая рекомендация

Для первой рабочей версии рекомендуется следующий стек:

- платформа: `.NET Framework 4.8`;
- UI: `WinForms`;
- язык: `C#`;
- модель интеграции с оборудованием: `COM / Automation / vendor SDK`;
- модель многопоточности: `Task + BlockingCollection + dedicated worker threads`;
- экспорт `XLSX`: `Open XML SDK` в потоковом режиме записи;
- логирование: встроенный файловый лог + журнал событий в UI;
- конфигурация: `app.config` + пользовательские настройки.

Это не самый "модный" стек, но это самый безопасный старт для системы, которая:

- в первую очередь должна стабильно работать на Windows 10/11;
- желательно может запускаться на Windows 7;
- должна работать под Windows;
- зависит от старых vendor API;
- использует COM / Automation;
- должна быть устойчивой при длительном измерении;
- должна быстро дать рабочий результат без войны с совместимостью драйверов.

## 2. Почему именно такой стек

### 2.1. Почему не .NET 8 как основной вариант

Теоретически приложение можно писать на современном `.NET`, но в данном проекте это не лучший старт из-за следующих рисков:

- часть оборудования и примеров поставляется в виде старых `COM / ActiveX` интеграций;
- примеры из архива ориентированы на классический Windows-interop;
- неизвестна разрядность и особенности регистрации библиотек;
- старые драйверы и automation API обычно проще и стабильнее заводятся в `.NET Framework`.

Если начать с `.NET 8`, можно потерять много времени на:

- разрядность процесса;
- регистрацию COM-компонентов;
- interop-обертки;
- различия поведения старых automation-сценариев.

Для измерительного ПО это лишний риск.

### 2.2. Почему `.NET Framework 4.8`

`.NET Framework 4.8` подходит лучше всего, потому что:

- это нативная и ожидаемая среда для старого Windows COM interop;
- это хороший компромисс для Windows 10/11 и возможной совместимости с Windows 7;
- она хорошо совместима с примерами от вендора;
- она упрощает подключение библиотек из `Beam Gage` и `Ophir`;
- она снижает риск проблем с `ActiveX`, `COM registration` и `x86/x64`;
- для простой операторской утилиты ее возможностей более чем достаточно.

### 2.3. Почему `WinForms`

Для первой версии рекомендуется `WinForms`, а не `WPF`, потому что:

- нужен простейший интерфейс, а не сложный rich-client UI;
- `WinForms` быстрее поднять;
- vendor-примеры уже даны на `WinForms`-подобной модели;
- проще интегрировать старые .NET / COM сценарии;
- ниже стоимость отладки UI-потоков и сообщений оператора.

Если позже понадобится более сложный интерфейс, архитектуру можно сохранить, а UI заменить.

## 3. Целевая разрядность

Для проекта нужно сразу заложить явную разрядность сборки.

Рекомендация для старта:

- основная цель сборки: `x86`.

Причина:

- очень часто старые `COM` библиотеки и vendor automation SDK поставляются как `32-bit`;
- в архиве уже встречаются признаки x86-ориентированных компонентов и примеров;
- `x86` как стартовая цель минимизирует риск "Class not registered" и подобных ошибок.

После уточнения версий драйверов и библиотек можно отдельно проверить:

- возможна ли `x64` сборка;
- нужен ли второй профиль сборки;
- можно ли безопасно перейти на `AnyCPU`.

Но первая промышленно полезная версия должна проектироваться под один проверенный target.

## 4. Рекомендуемая архитектура

Рекомендуется модульный монолит с четким разделением на слои.

### 4.1. Слои

#### UI Layer

Отвечает за:

- экран оператора;
- ввод параметров;
- отображение текущих значений;
- отображение состояния подключения;
- отображение ошибок и событий;
- запуск и остановку серии.

UI не должен содержать логику измерений или синхронизации.

#### Application Layer

Отвечает за:

- оркестрацию сценария измерения;
- управление жизненным циклом серии;
- координацию модулей acquisition, processing и export;
- реакцию на ошибки.

Это главный "дирижер" приложения.

#### Device Integration Layer

Отвечает за:

- работу с `Beam Gage`;
- работу с `Ophir / Pulsar-4`;
- инициализацию, старт, чтение данных, останов;
- преобразование vendor-данных в единый формат приложения.

#### Synchronization Layer

Отвечает за:

- сопоставление измерений двух трактов;
- временную привязку событий;
- фиксацию рассинхронизации;
- выдачу синхронных пар дальше в обработку.

#### Processing Layer

Отвечает за:

- ведение буферов по последним `N` измерениям;
- вычисление скользящих средних;
- определение стационарного режима;
- определение выхода из стационарного режима.

#### Persistence Layer

Отвечает за:

- запись сырых данных;
- запись событий;
- формирование `XLSX`;
- сохранение логов.

## 8. Модель многопоточности

Для устойчивой работы нужно не смешивать поток UI и поток измерений.

Рекомендуемая схема:

- отдельный поток или callback chain для `Beam Gage`;
- отдельный поток или callback chain для `Ophir / Pulsar-4`;
- общая thread-safe очередь на вход в синхронизатор;
- отдельный worker для синхронизации;
- отдельный worker для записи в `XLSX`;
- UI обновляется только через marshaling в UI thread.

### 8.1. Почему не писать сразу в UI и Excel из callback

Это плохая идея, потому что:

- callback от оборудования должен освобождаться быстро;
- запись в `XLSX` относительно тяжелая;
- UI может подвисать;
- будут трудноуловимые race conditions.

Правильный подход:

- быстро принять sample;
- положить его в очередь;
- обрабатывать дальше в фоновых конвейерах.

### 8.2. Рекомендуемый примитив очередей

Если остается `.NET Framework 4.8`, то достаточно:

- `BlockingCollection<T>`;
- `ConcurrentQueue<T>`;
- `CancellationToken`;
- `Task.Factory.StartNew(..., TaskCreationOptions.LongRunning)`.

Этого достаточно для промышленно полезной первой версии.

## 9. Стратегия синхронизации

Поскольку оба тракта живут независимо, синхронизация должна быть явным компонентом.

### 9.1. Предпочтительный приоритет синхронизации

Приоритет сопоставления должен быть таким:

1. если API дает номер импульса или sequence id, использовать его;
2. если API дает собственную временную отметку измерения, использовать ее;
3. если ничего надежнее нет, использовать локальную monotonic timestamp на границе получения sample.

### 9.2. Механизм

Синхронизатор должен:

- держать отдельные короткие буферы по каждому источнику;
- искать ближайшую допустимую пару;
- если пара найдена, выдавать `SynchronizedMeasurementPair`;
- если пара не найдена за пределами окна, генерировать событие рассинхронизации.

### 9.3. Настраиваемый параметр

Нужно заложить параметр:

- `MaxSynchronizationDeltaMs`.

## 10. Алгоритм определения стационарности

Так как заказчик не фиксирует конкретную метрику, лучше сделать заменяемую политику.

### 10.1. Рекомендуемая первая реализация

Для первой версии предлагается:

- брать последние `N` синхронных пар;
- считать скользящее среднее отдельно по каждому тракту;
- сравнивать текущее среднее с предыдущим окном;
- рассчитывать относительное отклонение;
- считать режим стационарным, если отклонение ниже порога;
- считать выход из стационарности, если отклонение стало выше порога выхода.

### 10.2. Почему именно так

Потому что этот алгоритм:

- прост;
- прозрачен;
- легко объясним оператору;
- легко тестируется;
- легко заменяется позже.

### 10.3. Что стоит заложить в коде

Нужно сделать интерфейс политики:

```csharp
public interface IStationarityPolicy
{
    StationarityUpdate Evaluate(
        IReadOnlyList<double> beamValues,
        IReadOnlyList<double> ophirValues,
        StationarityContext context);
}
```

Это позволит потом заменить критерий без ломки приложения.

## 11. Запись в `XLSX`

### 11.1. Почему не `ClosedXML` как основной вариант

`ClosedXML` очень удобен, но для длинных серий и большого числа строк может:

- держать слишком много данных в памяти;
- замедляться на больших объемах.

При `1000 Гц` и серии до `10 минут` поток может быть большим, поэтому безопаснее использовать потоковую запись.

### 11.2. Рекомендуемый вариант

Использовать:

- `DocumentFormat.OpenXml`;
- `OpenXmlWriter` для потоковой записи.

Это даст:

- лучший контроль памяти;
- устойчивость на больших объемах;
- честный `XLSX`.

### 11.3. Файл результатов

Рекомендуемая структура:

- `RawData`;
- `Events`;
- `Summary`.

При необходимости позже можно добавить:

- `Settings`;
- `Diagnostics`.

- операторский журнал в UI;

### 12.1. Что писать в технический лог

- старт приложения;
- версии библиотек;

- параметры `N`, порогов и окна синхронизации;
- события стационарности;
- события рассинхронизации;
- ошибки устройства;
- ошибки записи файла.

## 13. Конфигурация

В первой версии достаточно:

- `app.config` для системных настроек;
- пользовательских настроек для последнего пути экспорта и последних параметров.

Нужно хранить:

- последнее значение `N`;
- последний порог стационарности;
- последнее окно синхронизации;
- последний путь сохранения;
- признак автосохранения настроек.

## 14. План по устойчивости к ошибкам

### 14.1. Ошибки устройства

Каждый адаптер устройства должен:

- сам уметь перевести vendor-ошибку во внутренний тип ошибки;
- генерировать `Faulted`;
- не падать молча.

### 14.2. Ошибка одного тракта

Если любой тракт падает критически:

- сессия завершается аварийно;
- оператор получает сообщение;
- экспорт корректно закрывается;
- лог фиксирует причину.

### 14.3. Ошибка экспорта

Если `XLSX` временно недоступен или поврежден:

- нужно завершить сессию как ошибочную;
- показать оператору причину;
- по возможности сохранить аварийный лог.

## 15. Тестовая стратегия

Даже при работе с железом нужно заложить слой тестируемости.

### 15.1. Что можно тестировать без оборудования

- синхронизатор;
- буферы последних `N`;
- алгоритм стационарности;
- реакцию на рассинхронизацию;
- экспорт `XLSX`;
- orchestration сессии.

### 15.2. Как это сделать

Нужны fake-источники:

- `FakeMeasurementSource("BeamGage")`;
- `FakeMeasurementSource("Ophir")`.

Они должны уметь:

- генерировать синхронные потоки;
- генерировать рассинхрон;
- генерировать пропуск импульса;
- генерировать резкий выход из стационарности;
- генерировать device fault.

## 16. Альтернативы и компромиссы

### 16.1. Возможная альтернатива 1

`WPF + .NET Framework 4.8`

Подходит, если:

- нужен более аккуратный UI;
- в команде есть опыт с `MVVM`;
- время на интерфейс не критично.

Но для первого релиза это не дает заметной пользы по сравнению с `WinForms`.

### 16.2. Возможная альтернатива 2

`.NET 8 + WinForms`

Подходит, только если после проверки окажется, что:

- все нужные SDK стабильно работают;
- нет проблем с COM registration;
- нет ограничений по разрядности.

До этой проверки это лишний риск.

Дополнительно этот вариант хуже подходит, если нужно сохранить шанс совместимости с Windows 7.

### 16.3. Возможная альтернатива 3

`Python`

Для этого проекта не рекомендуется как основной стек, потому что:

- Windows COM/driver integration будет менее предсказуемой;
- сложнее упаковать и передать оператору;
- выше риск runtime-проблем на рабочем месте;
- слабее история с vendor-примерами.

## 17. Итоговое решение

Для старта рекомендую зафиксировать:

- `C#`;
- `.NET Framework 4.8`;
- `WinForms`;
- `x86`;
- `COM / Automation integration`;
- модульный монолит;
- фоновые очереди и worker-процессы;
- `Open XML SDK` для потокового `XLSX`.

Это даст лучший баланс между:

- скоростью разработки;
- совместимостью с архивными примерами;
- совместимостью с Windows 10/11 и потенциально с Windows 7;
- устойчивостью на Windows;
- предсказуемостью поведения при работе с оборудованием.

## 18. Политика поддержки ОС

Рекомендуется зафиксировать следующую платформенную политику:

- основной поддерживаемый контур: Windows 10/11;
- дополнительный контур без обязательств на старте: Windows 7;
- подтверждение Windows 7 возможно только после проверки реальных драйверов, COM-компонентов и vendor SDK на тестовой машине.

## 19. Что я бы делал следующим шагом

Следующий практический шаг:

1. зафиксировать стек как baseline;
2. оформить схему solution и список проектов;
3. проверить реальные библиотеки `Beam Gage` и `Ophir` на разрядность и способ подключения;
4. собрать технический spike:
   - подключение к Beam Gage;
   - подключение к Ophir / Pulsar-4;
   - получение 1-2 тестовых измерений;
5. только после этого строить полноценный UI и экспорт.
