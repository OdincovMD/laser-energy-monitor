# Инструкция для заказчика: первый запуск проверки Ophir

## Цель

Эта проверка нужна, чтобы подтвердить:

- установлен ли на компьютере необходимый Ophir runtime;
- видит ли приложение библиотеку Ophir COM;
- видит ли система подключённое устройство по USB;
- можно ли переходить к проверке живого потока измерений.

## Что должно быть подготовлено заранее

Перед запуском проверки просим убедиться, что:

- установлен vendor runtime / automation package для Ophir;
- устройство Ophir подключено по USB;
- датчик и контроллер включены и готовы к работе;
- никакая другая программа не удерживает устройство открытым;
- если в вашей стандартной схеме работы используется фирменное ПО Ophir, оно либо закрыто, либо не блокирует доступ к устройству.

## Что нужно сделать

### 1. Запустить приложение

Запустите приложение `Laser Energy Monitor`.

### 2. Открыть блок диагностики

В левой части окна находится блок `Session Control`.

Для первичной проверки используйте:

- `Run Self-Test`
- `Ophir Smoke-Test`

### 3. Выполнить Hardware Self-Test

Нажмите кнопку `Run Self-Test`.

Эта проверка показывает:

- доступен ли COM runtime;
- может ли приложение активировать Ophir COM object;
- возвращает ли `ScanUSB` подключённые устройства;
- есть ли активный датчик;
- можно ли перейти к краткой stream-проверке.

### 4. Выполнить Ophir Smoke-Test

Нажмите кнопку `Ophir Smoke-Test`.

Эта проверка запускает короткий тест Ophir SDK и формирует отдельный текстовый отчёт.

## Какой результат считается нормальным

### Вариант A: runtime есть, но устройство не подключено

Нормальным считается результат, в котором:

- COM registration = `PASS`
- COM activation = `PASS`
- USB scan показывает `Detected devices: 0`
- итоговый результат содержит смысл:
  `Runtime available, but no USB devices are currently visible. Acquisition was skipped.`

Это означает, что программная часть установлена корректно, но физическое устройство не обнаружено.

### Вариант B: устройство подключено и доступно

Нормальным считается результат, в котором:

- USB scan находит хотя бы одно устройство;
- проверка открытия устройства проходит успешно;
- проверка датчика проходит успешно;
- smoke-test получает хотя бы несколько sample-значений;
- в отчёте нет критической fault-ошибки.

## Какие файлы нужно передать нам после проверки

Просим прислать следующие материалы:

- полный текст окна `Hardware Self-Test`;
- полный текст окна `Ophir Smoke-Test`;
- файл `application.log`;
- файл вида `hardware-self-test-*.txt`;
- файл вида `ophir-smoke-test-*.txt`;
- если появились файлы в папке `ophir-captures`, приложить их тоже.

## Где обычно находятся отчёты

Если приложение запускалось из стандартной сборки, файлы обычно находятся рядом с приложением в подпапке:

- `output`

В этой папке могут появиться:

- `application.log`
- `hardware-self-test-*.txt`
- `ophir-smoke-test-*.txt`
- папка `ophir-captures`

## Что дополнительно сообщить вместе с отчётом

Просим также указать:

- модель устройства Ophir;
- модель датчика;
- версию установленного vendor runtime / automation package;
- версию Windows;
- видно ли устройство в фирменном ПО Ophir;
- была ли на момент проверки открыта какая-либо другая программа, работающая с устройством.

## Если проверка завершилась ошибкой

Если возникла ошибка, просим:

- не повторять сразу много разных действий подряд;
- сохранить текст ошибки;
- сохранить отчёты и `application.log`;
- сообщить, на каком именно шаге возникла проблема:
  `Run Self-Test` или `Ophir Smoke-Test`;
- сообщить, видит ли устройство фирменное ПО Ophir.

## Что будет после этой проверки

Если runtime и устройство определяются корректно, следующим шагом будет:

- подтверждение живого получения sample-данных;
- проверка channel selection;
- уточнение timestamp strategy;
- финальная доводка под ваш реальный стенд.
