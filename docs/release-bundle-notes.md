# Laser Energy Monitor: комплект поставки

## Что входит в комплект поставки

В комплект поставки входит папка с `Release`-сборкой приложения `Laser Energy Monitor` для `.NET Framework 4.8`.

Основные файлы:

- `LaserEnergyMonitor.Wpf.exe`
- `LaserEnergyMonitor.Wpf.exe.config`
- библиотеки `LaserEnergyMonitor.*.dll`
- библиотеки `DocumentFormat.OpenXml*.dll`
- эта памятка
- инструкция по первичному запуску `Ophir`

## Что нужно для запуска

Минимально:

- Windows с установленным `.NET Framework 4.8`
- права на чтение и запись в папку, где лежит приложение

Для работы только в режиме имитации:

- дополнительные runtime от вендора не требуются

Для работы с реальным `BeamGage`:

- установлен `BeamGage Professional` с поддержкой automation
- доступны сборки вендора `Spiricon.Automation.dll` и `Spiricon.BeamGage.Automation.dll`
- установленный runtime совместим с `x86`

Для работы с реальным `Ophir`:

- установлен пакет automation / COM runtime от вендора
- в системе зарегистрирован `OphirLMMeasurement.CoLMMeasurement`
- установленный runtime совместим с `x86`
- устройство видно по USB и не занято другой программой

## Как запускать

1. Распаковать комплект поставки в локальный каталог, где у пользователя есть право записи.
2. Запустить `LaserEnergyMonitor.Wpf.exe`.
3. При первом запуске проверить выбранные источники данных в блоке `Run Setup`.

По умолчанию приложение может работать в безопасном режиме с симулированными источниками:

- `Simulated BeamGage`
- `Simulated Ophir`

## Что приложение создает рядом с собой

При запуске приложение использует подпапку `output`.

В ней могут появляться:

- `application.log`
- `measurement-session.xlsx`
- `measurement-session.RawData.csv`
- `measurement-session.Events.csv`
- `measurement-session.Summary.csv`
- `measurement-session.Stationary.csv`
- `hardware-self-test-*.txt`
- `beamgage-smoke-test-*.txt`
- `ophir-smoke-test-*.txt`
- подпапка `ophir-captures` при включенном захвате сырых данных `Ophir`

Важно:

- итоговый `.xlsx` создается приложением автоматически;
- вспомогательные CSV-файлы остаются рядом и используются как диагностические артефакты;
- для записи результатов у пользователя должен быть доступ на запись в каталог комплекта поставки или в явно заданный `Output Path`.

## Что уже проверено в разработке

Подтверждено:

- `Debug` и `Release` сборки проходят без ошибок;
- unit-тесты проходят: `8/8`;
- доступны диагностические сценарии:
  `Self-Test`, `BeamGage Test`, `Ophir Smoke-Test`.

Не подтверждено без живого стенда:

- полноценная работа `BeamGage` на целевой машине;
- полноценная работа `Ophir / Pulsar-4` на целевой машине;
- длительная реальная серия с обоими приборами одновременно.

## Проверка на целевой машине

Перед финальной приемкой выполнить:

1. Запуск приложения на целевой машине.
2. Проверку `Self-Test`.
3. Проверку `BeamGage Test`, если используется реальный `BeamGage`.
4. Проверку `Ophir Smoke-Test`, если используется реальный `Ophir`.
5. Короткую реальную сессию измерения с сохранением результатов.

## Если планируется работа с реальным Ophir

Используйте отдельную инструкцию:

- `ophir-first-run-validation.md`

В ней описано:

- что должно быть подготовлено заранее;
- как выполнить первую диагностику;
- какие файлы нужно вернуть после проверки.

## Ограничения текущей поставки

- приложение собрано под `x86`, поэтому runtime от вендора тоже должен быть совместим по разрядности;
- локальная сборка и unit-тесты не заменяют проверку на реальном стенде;
- итоговая готовность к промышленной эксплуатации зависит от реальной проверки `BeamGage` и `Ophir` на целевой машине.
